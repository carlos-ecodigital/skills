## Routing Rules

### Priority 1: Exact Skill Match
If the user's request directly names a skill or uses a registered trigger keyword from the table above, route to that skill immediately.

### Priority 2: Audience + Domain Match
If no exact match, determine:
1. **Who is the audience?** (investor, grower, supplier, gemeente, customer, internal)
2. **What domain?** (permitting, finance, technical, etc.)
3. Route to the skill matching both tags. If multiple match, prefer higher Tier.

### Priority 3: Function Match
If audience + domain doesn't resolve, determine the function:
- Asking a question? вҶ’ Q&A skills
- Need a document? вҶ’ generation skills
- Need a status update? вҶ’ tracking skills
- Need to evaluate something? вҶ’ analysis skills
- Complex multi-step task? вҶ’ task-executioner

### Priority 4: Orchestration
If the task spans multiple domains or requires multiple skills:
1. Route to `task-executioner` for decomposition
2. Task-executioner assigns sub-tasks to individual skills via RISE framework
3. `delegation-engine` routes execution to team members

### Conflict Resolution
| Conflict | Resolution |
|----------|-----------|
| Two skills claim same domain | Higher Tier wins |
| Question spans two domains | task-executioner decomposes |
| Audience unclear | Ask user: "Is this for [audience A] or [audience B]?" |
| No skill exists | Route to `forge` to assess if a new skill is needed |

### Skill Loading Protocol
When routing to a skill, load in this order (respect token budget):
1. `SKILL.md` вҖ” always load (operational instructions)
2. `identity.md` вҖ” load for voice/tone calibration
3. `principles.md` вҖ” load for decision framework
4. `soul.md` вҖ” load only when voice fidelity is critical (grower comms, investor docs)

See `skills/_retrieval-rules.yaml` for token-budget-aware context loading.
